系統名稱與概述

系統定義、目的、使用者角色與系統範疇

1.1 系統名稱

中文名稱飲食與熱量紀錄系統
英文名稱Daily Diet Accounting System（DDAS）
系統版本v2.0.0
系統類型純前端單頁應用程式（SPA）
部署方式靜態網頁，無需後端伺服器，使用 Vite 建置

1.2 系統背景與目的

本系統為淡江大學 JavaScript 網頁設計課程之期末專題，旨在提供使用者一個便利的飲食記錄與熱量追蹤工具。 系統完全執行於瀏覽器端，利用 localStorage 儲存所有資料，不依賴後端 API 或資料庫伺服器。

核心目標
  • 提供直覺的每日 4 餐（早/午/晚/點心）飲食紀錄功能
  • 追蹤熱量及三大巨量營養素（蛋白質、脂肪、碳水化合物）
  • 根據個人身體數據（BMR）計算每日目標熱量
  • 透過圖表視覺化呈現週/月飲食趨勢
  • 提供管理員後台維護食物資料庫與用戶帳號

1.3 使用者角色

👤 一般使用者（User）

  • 自行註冊帳號
  • 記錄每日飲食攝取
  • 查看個人營養摘要與趨勢
  • 設定個人身體數據與飲食目標
  • 瀏覽食物資料庫

🛡️ 系統管理員（Admin）

  • 擁有一般使用者全部權限
  • 管理食物資料庫（新增/編輯/刪除/CSV 匯入）
  • 查看所有使用者飲食紀錄
  • 管理用戶帳號（啟用/停用）
  • 發布系統公告
  • 查看平台統計數據

1.4 系統範疇

本系統為純前端應用，所有資料儲存於瀏覽器 localStorage，資料生命週期為 7 天（自動過期清除）。系統不包含後端 API、雲端同步或跨裝置資料共享功能。
功能範疇包含不包含
資料儲存瀏覽器 localStorage雲端資料庫、後端 API
身份驗證前端 Email/密碼驗證OAuth、SSO、多因素驗證
資料同步同一瀏覽器內跨裝置、跨瀏覽器同步
付款功能（無）不在範疇內
行動應用響應式網頁（RWD）原生 iOS/Android App

功能性需求

系統各模組的詳細功能規格，標示優先等級

優先等級說明： 必要 系統核心功能，版本一定須實作 ／ 建議 重要功能，應盡量實作 ／ 選用 增強功能，有餘力再實作

2.1 身份驗證模組

  • FR-A01使用者可使用 Email 與密碼進行帳號註冊，系統需驗證 Email 格式正確性必要
  • FR-A02密碼需長度至少 6 字元，且包含大寫、小寫、數字、特殊符號中至少 3 種字元類型必要
  • FR-A03使用者暱稱長度須介於 1 至 30 字元之間必要
  • FR-A04同一 Email 不可重複註冊，系統應即時提示必要
  • FR-A05使用者可使用 Email 與密碼登入，系統以 bcrypt 雜湊比對密碼必要
  • FR-A06登入失敗累計 5 次後，帳號鎖定 15 分鐘，並顯示剩餘等待時間必要
  • FR-A07被管理員停用的帳號無法登入，系統顯示「帳號已停用，請聯繫管理員」必要
  • FR-A08使用者可登出，系統清除 Session 並導向登入頁面必要
  • FR-A09Session 以 localStorage 儲存，7 天後自動過期需重新登入建議

2.2 飲食紀錄模組（Dashboard）

  • FR-R01使用者可新增飲食紀錄，欄位包含：餐別（早/午/晚/點心）、食物名稱、份量、熱量、蛋白質、脂肪、碳水化合物、備註必要
  • FR-R02新增時支援食物名稱關鍵字自動補全，從食物資料庫模糊比對，顯示前 5 筆候選結果必要
  • FR-R03選取食物資料庫中的食物後，依輸入份量自動換算所有營養素數值必要
  • FR-R04支援手動輸入食物（不選食物資料庫），使用者自行填入各營養素數值必要
  • FR-R05使用者可編輯已存在的飲食紀錄，修改任意欄位後儲存必要
  • FR-R06使用者可刪除飲食紀錄，需顯示確認對話框必要
  • FR-R07系統依日期顯示飲食紀錄，預設顯示當天必要
  • FR-R08提供「前一天 / 今天 / 下一天」快速切換與日期選擇器，可查看任意日期紀錄必要
  • FR-R09每日摘要自動計算並顯示當日總熱量、蛋白質、脂肪、碳水化合物攝取量必要
  • FR-R10熱量以甜甜圈圖呈現已攝取 vs 剩餘目標，三大營養素以進度條呈現完成百分比建議

2.3 食物資料庫模組（Food Search）

  • FR-F01使用者可瀏覽食物資料庫，每頁顯示 12 筆，提供分頁導覽必要
  • FR-F02支援以食物名稱關鍵字模糊搜尋必要
  • FR-F03支援依食物類別篩選（如：主食、蛋白質、蔬菜、飲品等）建議
  • FR-F04支援依熱量範圍（最低/最高）篩選食物選用
  • FR-F05每筆食物顯示：食物名稱、類別、每份熱量、蛋白質、脂肪、碳水化合物必要
  • FR-F06食物 ID 格式為 F001、F002… 依序遞增自動產生必要

2.4 個人設定模組（Settings）

  • FR-P01使用者可設定個人生理資料：性別、年齡、身高（cm）、體重（kg）必要
  • FR-P02使用者可設定活動量等級（久坐/輕度/中度/高度/非常高度）必要
  • FR-P03使用者可設定飲食目標（減重 ×0.8 / 維持 ×1.0 / 增重 ×1.15）必要
  • FR-P04系統依 Mifflin-St Jeor 公式計算 BMR，再乘以活動量與飲食目標係數，自動建議每日目標熱量建議
  • FR-P05使用者可手動覆蓋目標熱量及三大營養素目標值建議
  • FR-P06使用者可修改顯示暱稱選用
BMR 計算公式
性別公式（Mifflin-St Jeor）
男性88.362 + (13.397 × 體重) + (4.799 × 身高) − (5.677 × 年齡)
女性447.593 + (9.247 × 體重) + (3.098 × 身高) − (4.330 × 年齡)

最終目標熱量 ＝ BMR × 活動量係數 × 飲食目標係數

2.5 趨勢報表模組（Report）

  • FR-T01提供近 7 天及近 30 天兩種報表區間切換必要
  • FR-T02以折線圖/長條圖呈現每日熱量攝取趨勢，並標示目標熱量基準線必要
  • FR-T03以堆疊圖呈現每日三大營養素（蛋白質、脂肪、碳水化合物）分佈建議
  • FR-T04顯示區間內平均每日熱量、達標天數統計建議
  • FR-T05無紀錄的日期以 0 值顯示，不略過必要

2.6 管理員後台模組（Admin）

  • FR-AD01管理員儀表板顯示：活躍用戶數、今日/本週紀錄總數、食物資料庫大小、本週 Top 10 最常使用食物（圖表）必要
  • FR-AD02食物管理：新增、編輯、刪除食物（刪除前檢查是否有紀錄引用，若有則拒絕）必要
  • FR-AD03食物管理：支援 CSV 批量匯入，格式為 foodName, category, servingSize, servingUnit, calories, protein, fat, carb，逐行驗證並回報成功/失敗筆數建議
  • FR-AD04紀錄管理：可依日期範圍、用戶、餐別篩選，查看所有使用者飲食紀錄（含用戶名稱與 Email）必要
  • FR-AD05用戶管理：查看用戶列表、啟用/停用帳號，不可停用自己的帳號必要
  • FR-AD06公告管理：新增、編輯、刪除公告，設定標題、內容、有效日期範圍、啟用狀態建議

2.7 系統公告模組

  • FR-N01啟用中且在有效日期範圍內的公告，在 Dashboard 頂部以橫幅顯示建議
  • FR-N02使用者可點擊關閉按鈕隱藏當次 Session 的公告建議
  • FR-N03超過有效日期或 isActive=false 的公告不顯示必要

UI/UX 設計要求

使用者介面、互動體驗、視覺化元件規範

3.1 整體設計原則

  • UI-G01採用 Bootstrap 5 作為 UI 框架，確保跨瀏覽器一致的視覺呈現
  • UI-G02支援響應式設計（RWD），頁面需在桌面（≥768px）與行動裝置（<768px）皆可正常使用
  • UI-G03所有操作反饋（新增/修改/刪除/錯誤）以 Toast 通知呈現，自動於 3 秒後消失
  • UI-G04破壞性操作（刪除）必須顯示確認對話框（Bootstrap Modal），避免誤操作
  • UI-G05表單欄位需提供即時驗證提示，不合法的輸入應顯示紅色錯誤說明

3.2 版面配置

裝置寬度版面結構導覽方式

平板（768px–991px） 折疊式 Sidebar（可手動展開）+ 主內容區 Sidebar 收合按鈕 + 底部導覽列
手機（<768px） 全寬主內容區，Sidebar 隱藏 固定底部導覽列（4個主要頁面）
  • UI-L01Header 顯示系統名稱 LOGO、當前用戶暱稱、通知圖示
  • UI-L02行動裝置底部導覽列固定顯示：首頁、食物搜尋、報表、設定
  • UI-L03管理員後台採用獨立版面，與一般用戶頁面有明顯視覺區分

3.3 互動設計

  • UI-I01新增/編輯飲食紀錄透過 Bootstrap Modal 呈現，不跳頁
  • UI-I02食物名稱輸入框支援 debounce 自動補全下拉選單，選取後自動填入營養數值
  • UI-I03日期切換按鈕（← →）點擊後即時更新當日紀錄列表，無需重新整理
  • UI-I04食物搜尋頁面篩選條件變更時，自動重新篩選並更新分頁
  • UI-I05表單送出後，Modal 自動關閉並顯示成功 Toast

3.5 無障礙設計

  • UI-A01互動元素（按鈕、輸入框、連結）需加上語意化 aria-label 屬性
  • UI-A02Toast 通知區域加上 aria-live="polite"，確保螢幕閱讀器可讀取
  • UI-A03使用語意化 HTML 標籤（<main>、<nav>、<header>、<section>）
  • UI-A04支援鍵盤導覽（Tab 鍵）操作所有互動元件
  • UI-A05圖示搭配文字說明，不單獨依賴顏色傳遞資訊

資安要求

密碼安全、存取控制、輸入驗證、資料保護

本系統為純前端教學專題，資安措施以前端可實作的範圍為主。正式生產環境應搭配後端 API 與 HTTPS，以獲得完整安全保障。

5.1 密碼安全

  • SEC-P01密碼絕對不得以明文儲存，必須使用 bcryptjs 進行單向雜湊，salt rounds 設定為 12
  • SEC-P02密碼長度下限 6 字元，且需包含大寫字母、小寫字母、數字、特殊符號中至少 3 種
  • SEC-P03Session 物件（ddas_session）僅儲存 userId、displayName、email、role，不含密碼雜湊
  • SEC-P04登入時使用 bcrypt.compare() 比對，不以字串直接比較密碼

5.2 存取控制

  • SEC-AC01未登入用戶存取受保護路由時，系統自動導向 /login
  • SEC-AC02一般用戶存取 /admin/* 路由時，系統自動導向 /dashboard
  • SEC-AC03已登入用戶存取 /login 或 /register 時，系統自動導向 /dashboard（防止重複登入）
  • SEC-AC04飲食紀錄 Service 層以 userId 為範疇隔離資料，不允許跨用戶存取他人紀錄
  • SEC-AC05管理員停用帳號後，該帳號下次嘗試登入時立即拒絕，不等 Session 過期
  • SEC-AC06管理員不可停用自己的帳號，避免產生無法管理的系統狀態

5.3 輸入驗證與 XSS 防護

  • SEC-I01所有使用者輸入的文字欄位（食物名稱、備註、暱稱）在儲存前需通過 sanitizeInput() 過濾，移除所有 HTML 標籤（防止 XSS 攻擊）
  • SEC-I02Email 格式需符合正則表達式 /^[^\s@]+@[^\s@]+\.[^\s@]+$/ 才予以接受
  • SEC-I03數值型欄位（熱量、蛋白質、份量等）需確保為 Number 型別，避免字串注入
  • SEC-I04CSV 匯入時逐行驗證必要欄位，缺少欄位的資料列不匯入並回報錯誤
  • SEC-I05食物名稱不可為空字串，後端需驗證至少 1 字元

5.4 資料保護

  • SEC-D01所有 localStorage 資料設定 7 天過期時間，到期後自動清除，避免無限期資料累積
  • SEC-D02登出操作需完整清除 ddas_session，不允許 Session 殘留
  • SEC-D03登入失敗次數紀錄（ddas_login_attempts_*）於成功登入後立即清除
  • SEC-D04刪除用戶相關資料時，需同步清除 ddas_records_{userId} 與 ddas_profile_{userId}，避免孤兒資料
  • SEC-D05食物資料庫有飲食紀錄引用時，禁止刪除該食物，以維護資料完整性
