第 01 節｜系統概述
================================================================================

系統名稱（中）：每日飲食記帳系統
系統名稱（英）：Daily Diet Accounting System
系統類型：響應式網頁應用程式（RWD Web Application）
系統語言：繁體中文介面
目標平台：現代瀏覽器（Chrome、Firefox、Edge、Safari 最新兩版）

系統功能概述：
本系統讓使用者透過網頁記錄每日三餐與點心的飲食明細，自動計算熱量（kcal）
與三大營養素（蛋白質 g、脂肪 g、碳水化合物 g），並以視覺化圖表呈現攝取量
與個人目標的差距。後台管理員可維護食物資料庫、查閱使用者統計、管理帳號。

主要模組：
  [M01] 使用者驗證模組（登入 / 登出 / 密碼管理）
  [M02] 每日飲食總覽模組
  [M03] 食物搜尋與新增模組
  [M04] 個人目標設定模組
  [M05] 趨勢報表模組
  [M06] 後台管理模組（需管理員權限）

第 02 節｜使用者角色（Actor）定義
================================================================================

角色 01：一般使用者（User）
  - 說明：已登入的一般用戶，可記錄自己的飲食、設定目標、查看報表。
  - 對應功能：M02、M03、M04、M05

角色 02：管理員（Admin）
  - 說明：系統後台管理人員，擁有所有使用者的讀取權限及系統設定權。
  - 對應功能：M06（含 M02–M05 的唯讀查閱）
  - 登入入口：/admin（與前台分離）

角色 03：訪客（Guest）
  - 說明：未登入用戶，只能查看系統簡介頁與登入 / 註冊頁面。
  - 對應功能：僅限 /login、/register、/about

第 03 節｜使用情境（Use Cases）
================================================================================

[UC-01] 使用者註冊
  前置條件：使用者尚未擁有帳號
  主要流程：
    1. 使用者填寫 Email、密碼、確認密碼、暱稱
    2. 系統驗證 Email 格式與密碼強度
    3. 系統建立帳號，寫入資料庫，發送驗證信（可為模擬）
    4. 導向登入頁，顯示「註冊成功，請登入」
  異常流程：
    - Email 已存在 → 顯示「此 Email 已被使用」
    - 密碼強度不足 → 顯示密碼規則說明

[UC-02] 使用者登入
  前置條件：使用者已完成註冊
  主要流程：
    1. 輸入 Email 與密碼
    2. 系統驗證帳密（後端 bcrypt 比對）
    3. 驗證通過 → 發放 JWT Token，存入 HttpOnly Cookie
    4. 導向今日飲食總覽頁（/dashboard）
  異常流程：
    - 帳密錯誤連續 5 次 → 帳號鎖定 15 分鐘

[UC-03] 新增飲食紀錄
  前置條件：使用者已登入
  主要流程：
    1. 使用者點擊「新增飲食」按鈕
    2. 輸入食物名稱（關鍵字自動補全，帶出資料庫資料）
    3. 選擇餐別（早餐／午餐／晚餐／點心）
    4. 輸入份量倍數，系統自動計算熱量與營養素
    5. 選填備註，點擊「儲存」
    6. 資料寫入，今日總覽即時更新
  異常流程：
    - 食物不在資料庫 → 使用者可手動輸入熱量與營養素

[UC-04] 設定每日目標
  前置條件：使用者已登入，尚未設定或需要修改目標
  主要流程：
    1. 進入「設定」頁面
    2. 輸入生理資料（性別、年齡、身高、體重、活動量）
    3. 選擇飲食目標（減重 / 維持 / 增肌）
    4. 系統用 Harris-Benedict 公式計算建議熱量
    5. 使用者可手動覆蓋目標值
    6. 儲存後立即反映於總覽頁的進度條
  Harris-Benedict 公式（供開發參考）：
    男性 BMR = 88.362 + (13.397 × 體重kg) + (4.799 × 身高cm) - (5.677 × 年齡)
    女性 BMR = 447.593 + (9.247 × 體重kg) + (3.098 × 身高cm) - (4.330 × 年齡)
    活動量乘數：久坐=1.2、輕度=1.375、中度=1.55、高度=1.725、非常高度=1.9
    飲食目標調整：減重 BMR×0.8、維持 BMR×1.0、增肌 BMR×1.15

[UC-05] 查看趨勢報表
  前置條件：使用者已有至少 1 天的飲食紀錄
  主要流程：
    1. 進入「報表」頁面
    2. 選擇時間範圍（7 天 / 30 天）
    3. 系統顯示熱量折線圖、營養素圓餅圖

[UC-06] 管理員管理食物資料庫
  前置條件：以管理員身份登入後台
  主要流程：
    1. 進入後台「食物管理」頁
    2. 可執行：新增食物、編輯食物、刪除食物、CSV 批次匯入
    3. 修改立即反映至前台搜尋結果

------------------------------------------------------------
[PAGE-01] 每日飲食總覽頁（/dashboard）
------------------------------------------------------------
URL：/dashboard
存取權限：需登入（一般使用者）

頁面元素與行為：

  [E-01] 日期切換器
    - 預設顯示今日（系統時間）
    - 提供「上一天」「下一天」按鈕
    - 提供日期選取器（date picker），點擊日期顯示該日紀錄
    - 不得選取「未來日期」

  [E-02] 熱量環形進度圖（Donut Chart）
    - 使用 Chart.js 繪製
    - 圓心顯示：已攝取 kcal / 目標 kcal
    - 顏色規則：
        < 75% 目標 → 綠色（#2e7d32）
        75%–100%  → 橙色（#e65100）
        > 100%    → 紅色（#c62828）
    - 即時更新（每次新增 / 刪除紀錄後重繪）

  [E-03] 三大營養素進度條
    - 蛋白質、脂肪、碳水化合物各一條水平進度條
    - 格式：「蛋白質 攝取g / 目標g」
    - 超過 100% 顯示紅色警示

  [E-04] 餐別分區列表
    - 分為「早餐」「午餐」「晚餐」「點心」四個 Accordion 或 Tab 區塊
    - 每筆紀錄顯示：食物名稱、份量、熱量（kcal）、三大營養素（g）、備註
    - 每筆紀錄提供「編輯」「刪除」操作按鈕
    - 刪除需彈出確認對話框（避免誤刪）

  [E-05] 快速新增按鈕
    - 固定在右下角或每個餐別分區的「+ 新增」按鈕
    - 點擊後進入新增飲食流程（參考 UC-03）

[PAGE-02] 食物搜尋頁（/food-search）

URL：/food-search

頁面元素：

[PAGE-03] 新增 / 編輯飲食紀錄 Modal
------------------------------------------------------------
呈現方式：Modal 彈窗（overlay），不跳轉頁面
存取權限：需登入

表單欄位定義（以下為完整欄位規格）：

  欄位名稱         | 型別              | 驗證規則                        | 必填
  ------------------|-------------------|----------------------------------|------
  mealType         | select (Enum)     | 值限 breakfast/lunch/dinner/snack | 必填
  recordDate       | date              | 不得為未來日期，格式 YYYY-MM-DD   | 必填
  foodName         | text (autocomplete)| 長度 1–50 字，禁止 HTML 標籤      | 必填
  servingAmount    | number            | 數值 > 0，最多 2 位小數           | 必填
  calories         | number            | 數值 >= 0，整數                   | 必填（可自動帶入）
  protein          | number            | 數值 >= 0，最多 1 位小數          | 選填
  fat              | number            | 數值 >= 0，最多 1 位小數          | 選填
  carbohydrate     | number            | 數值 >= 0，最多 1 位小數          | 選填
  note             | textarea          | 長度 0–200 字，禁止 HTML 標籤     | 選填

  自動計算邏輯：
    當使用者修改 servingAmount 時，系統自動重新計算：
      calories     = 單份熱量 × servingAmount（四捨五入至整數）
      protein      = 單份蛋白質 × servingAmount（四捨五入至小數第一位）
      fat          = 單份脂肪 × servingAmount（四捨五入至小數第一位）
      carbohydrate = 單份碳水 × servingAmount（四捨五入至小數第一位）

  送出成功後行為：
    1. Modal 關閉
    2. 今日總覽頁的熱量圓餅圖與進度條即時更新
    3. 對應餐別列表新增一筆紀錄
    4. 頁面頂端顯示 Toast 通知：「✓ 飲食紀錄已儲存，今日累計 XXX kcal。」
       Toast 顯示 3 秒後自動消失

------------------------------------------------------------
URL：/report

    - 使用者可手動記錄體重（每日一筆）
    - X 軸：日期，Y 軸：體重（kg）
    - 同時顯示 BMI 數值標籤

第 06 節｜資料結構定義（Data Schema）
================================================================================

------------------------------------------------------------
[SCHEMA-01] 飲食紀錄（dietRecord）
------------------------------------------------------------
所有欄位名稱採用小駝峰（lowerCamelCase）命名。

  欄位名稱       | 型別          | 長度限制  | 預設值            | 說明
  ----------------|---------------|-----------|-------------------|---------------------------
  recordId        | String (UUID) | 36 chars  | 系統自動產生 uuid | 唯一識別碼
  userId          | String        | 36 chars  | 來自登入 Token    | 使用者識別碼（外鍵）
  mealType        | String (Enum) | —         | —                 | breakfast/lunch/dinner/snack
  recordDate      | String        | 10 chars  | 今日日期          | 格式：YYYY-MM-DD
  foodName        | String        | 50 chars  | —                 | 食物名稱（可自訂）
  foodId          | String / null | 10 chars  | null              | 資料庫食物 ID；自訂食物為 null
  servingAmount   | Number        | —         | 1.0               | 份量倍數（小數點第二位）
  calories        | Number (int)  | —         | 0                 | 總熱量（kcal）
  protein         | Number        | —         | 0.0               | 蛋白質（g，小數第一位）
  fat             | Number        | —         | 0.0               | 脂肪（g，小數第一位）
  carbohydrate    | Number        | —         | 0.0               | 碳水化合物（g，小數第一位）
  note            | String        | 200 chars | ""（空字串）      | 備註
  createdAt       | String (ISO)  | —         | 系統當下時間      | 建立時間（UTC）
  updatedAt       | String (ISO)  | —         | 系統當下時間      | 最後修改時間（UTC）

------------------------------------------------------------
[SCHEMA-02] 食物資料（foodItem）
------------------------------------------------------------

  欄位名稱            | 型別         | 長度限制 | 說明
  ---------------------|--------------|----------|----------------------------------
  foodId               | String       | 10 chars | 唯一識別碼（如 "F001"）
  foodName             | String       | 50 chars | 食物名稱
  category             | String (Enum)| —        | 主食/蔬菜/水果/肉類/蛋類/乳製品/飲料/零食/其他
  servingSize          | Number       | —        | 每份基準重量或容量（數字）
  servingUnit          | String       | 20 chars | 份量單位顯示文字（如 "克（1 碗）"）
  caloriesPerServing   | Number (int) | —        | 每份熱量（kcal）
  proteinPerServing    | Number       | —        | 每份蛋白質（g）
  fatPerServing        | Number       | —        | 每份脂肪（g）
  carbPerServing       | Number       | —        | 每份碳水化合物（g）
  isCustom             | Boolean      | —        | false=系統資料，true=使用者自訂
  createdBy            | String       | 36 chars | 建立者 userId（系統預設資料填 "system"）
  createdAt            | String (ISO) | —        | 建立時間
  updatedAt            | String (ISO) | —        | 最後修改時間

[SCHEMA-03] 使用者設定（userProfile）
------------------------------------------------------------

  欄位名稱        | 型別          | 說明
  ----------------|---------------|------------------------------------------
  userId          | String (UUID) | 使用者識別碼（主鍵）
  displayName     | String        | 顯示名稱，長度 1–30
  email           | String        | 電子郵件（唯一）
  passwordHash    | String        | bcrypt hash，不得明文儲存
  gender          | String (Enum) | male / female
  age             | Number (int)  | 範圍 10–120
  heightCm        | Number        | 範圍 50.0–250.0
  weightKg        | Number        | 範圍 20.0–300.0

[TECH-01] 命名規範

     * 參數說明：
     *   @param {Array} records - 當日飲食紀錄陣列

[TECH-03] localStorage 7 天過期機制
  寫入範例（ES6 arrow function）：
    const setLocalStorageWithExpiry = (key, data) => {
      const item = {
        data,
        // 計算 7 天後的 ISO 時間字串作為到期時間
        expiredAt: new Date(Date.now() + 7 * 24 * 60 * 60 * 1000).toISOString()
      };
      localStorage.setItem(key, JSON.stringify(item));
    };

  讀取範例：
    const getLocalStorageWithExpiry = (key) => {
      const raw = localStorage.getItem(key);
      if (!raw) return null;
      const item = JSON.parse(raw);
      // 若已超過到期時間，清除並回傳 null
      if (new Date() > new Date(item.expiredAt)) {
        localStorage.removeItem(key);
        return null;
      }
      return item.data;
    };

[TECH-04] 禁止使用 for 迴圈，改用陣列方法
  - 禁止使用 for、for...in、for...of、while 處理陣列資料
  - 必須使用：map()、filter()、reduce()、find()、findIndex()、some()、every()
  - 範例（計算今日熱量總和）：
    // 錯誤寫法（禁止）：
    let total = 0;
    for (let i = 0; i < records.length; i++) { total += records[i].calories; }

    // 正確寫法：
    // 使用 reduce() 累加今日所有紀錄的熱量
    const totalCalories = records.reduce((sum, record) => sum + record.calories, 0);

第 12 節｜資安要求
================================================================================

[SEC-01] 密碼政策
  密碼需同時符合以下三項規則（從四種字元類型中選三種）：
    類型 A：英文大寫字母（A–Z）
    類型 B：英文小寫字母（a–z）
    類型 C：數字（0–9）
    類型 D：特殊符號（!@#$%^&*()_+-=[]{}|;':,./<>?）
  最短長度：6 個字元
  前端驗證正規表達式範例（強度計算）：
    const hasUpper   = /[A-Z]/.test(pwd);
    const hasLower   = /[a-z]/.test(pwd);
    const hasNumber  = /[0-9]/.test(pwd);
    const hasSymbol  = /[!@#$%^&*()_+\-=\[\]{};':,./<>?]/.test(pwd);
    const typeCount  = [hasUpper, hasLower, hasNumber, hasSymbol].filter(Boolean).length;
    const isValid    = pwd.length >= 6 && typeCount >= 3;
  後端同樣須執行相同驗證，前端驗證僅為輔助體驗。

[SEC-02] CSP 內容安全政策
  HTTP Response Header 設定：
    Content-Security-Policy:
      default-src 'self';
      script-src 'self' https://cdn.jsdelivr.net https://cdnjs.cloudflare.com
                        https://www.googletagmanager.com;
      style-src 'self' https://cdn.jsdelivr.net https://cdnjs.cloudflare.com;
      img-src 'self' data:;
      connect-src 'self' https://www.google-analytics.com;
      frame-ancestors 'none';
      base-uri 'self';
      form-action 'self';

[SEC-03] Cookie 安全設定
  JWT Token 存入 Cookie 時必須設定：
    HttpOnly : 前端 JavaScript 無法讀取（防 XSS 竊取）
    Secure   : 僅限 HTTPS 傳送
    SameSite : Strict（防 CSRF）
    Path     : /
    Max-Age  : 86400（1 天，單位：秒）
  後端設定範例（Node.js / Express）：
    res.cookie('authToken', jwtToken, {
      httpOnly: true,
      secure: process.env.NODE_ENV === 'production',
      sameSite: 'Strict',
      path: '/',
      maxAge: 86400 * 1000
    });

[SEC-04] 輸入驗證與注入防護
  前端規則：
    - 所有文字輸入欄位在插入 DOM 時使用 textContent，禁止 innerHTML
    - 表單送出前以正規表達式驗證格式，拒絕含有 <script>、<、> 的輸入
    - 數字欄位強制轉型 Number()，非數字顯示錯誤訊息
  後端規則：
    - 所有資料庫操作使用參數化查詢（Parameterized Query / Prepared Statement）
    - 禁止字串拼接 SQL
    - 輸入長度超過欄位定義上限時，回傳 400 Bad Request
    - 對 HTML 特殊字元進行跳脫（&lt; &gt; &amp; &quot; &#x27;）

[SEC-05] OWASP Top 10 2021 對應措施
  A01 權限控制失效    → 後端所有 API 驗證 JWT，後台 API 另驗 admin role
  A02 加密失敗        → 密碼 bcrypt（cost=12），傳輸使用 HTTPS，敏感欄位不入 Log
  A03 注入攻擊        → 參數化查詢，輸入驗證，輸出跳脫（見 SEC-04）
  A04 不安全設計      → 已定義角色分層（Guest/User/Admin），最小權限原則
  A05 安全設定錯誤    → CSP Header、X-Frame-Options: DENY、X-Content-Type-Options: nosniff
  A06 脆弱元件        → 前端 CDN 使用固定版本號，後端 npm audit 定期執行
  A07 驗證與授權失敗  → JWT 驗證、密碼 bcrypt、登入失敗鎖定（5次/15分鐘）
  A08 軟體與資料完整性→ SRI（Subresource Integrity）Hash 驗證 CDN 資源
  A09 安全日誌不足    → 後端記錄登入成功/失敗、API 錯誤、管理員操作 Log
  A10 SSRF            → 後端不接受使用者輸入的 URL 作為資料來源
