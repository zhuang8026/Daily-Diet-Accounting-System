# 每日飲食記帳系統（Daily Diet Accounting System, DDAS）技術簡報

---

## Slide 1｜專案介紹

**負責講者：Person A**

### 內容重點
- 專案名稱：每日飲食記帳系統（Daily Diet Accounting System, DDAS）
- 系統目的
  - 記錄每日三餐與飲食內容
  - 追蹤每日熱量攝取
  - 分析三大營養素：蛋白質、脂肪、碳水化合物
  - 協助使用者建立長期飲食管理習慣
- 使用者角色
  - 一般用戶：記錄、查詢、分析個人飲食資料
  - 管理員：管理食物資料、用戶帳號、公告與操作日誌
- 現行技術棧
  - Frontend：React 18 + Vite 5 + React Router v6 + Bootstrap 5.3 + Chart.js 4.4
  - Backend：FastAPI + Pydantic v2 + python-jose + passlib[bcrypt]
  - Auth：JWT + HttpOnly Cookie
  - Testing：Playwright E2E

### 講者稿
這個系統主要解決「每日飲食難以持續追蹤」的問題，讓使用者可以記錄三餐並查看營養趨勢。  
除了用戶側的飲食管理，我們也設計了管理員後台，負責維護食物資料、帳號與系統紀錄。  
目前系統採用 React 前端與 FastAPI 後端，並使用 JWT 搭配 HttpOnly Cookie 完成登入認證。

## Slide 10｜JWT + HttpOnly Cookie 認證機制與帳號鎖定

**負責講者：Person C**

### 內容重點
- 登入流程
  1. 前端送出 `POST /auth/login`
  2. 後端驗證 email / password
  3. 檢查帳號是否啟用與是否被鎖定
  4. 成功後簽發 JWT（HS256，24 小時有效）
  5. 後端透過 `response.set_cookie(httponly=True, samesite="lax")` 寫入 Cookie
  6. 後續請求由 axios `withCredentials: true` 自動帶出 Cookie
  7. 登出時呼叫 `POST /auth/logout`，後端刪除 Cookie
  8. 若 API 回傳 401，前端導向登入頁
- JWT payload
  - userId
  - email
  - role
  - exp
- 雙層授權
  - `get_current_user()`：解碼 Cookie 中的 JWT，確認使用者存在且啟用
  - `require_admin()`：額外確認角色是否為 admin
- 帳號鎖定
  - 登入失敗累計 5 次後鎖定 15 分鐘
  - 前端顯示鎖定提示
  - 鎖定時間到期後可重新登入

### 講者稿
最新版登入機制是由後端驗證帳密，並把 JWT 寫入 HttpOnly Cookie。  
前端不需要、也不能直接讀取 Token，這能降低 Token 被前端腳本竊取的風險。  
另外，登入失敗 5 次會鎖定帳號 15 分鐘，用來降低暴力破解風險。

## Slide 11｜資安設計 — 已實作防禦機制

**負責講者：Person C**

### 內容重點
- 密碼安全
  - passlib[bcrypt] 進行密碼雜湊
  - 不儲存明文密碼
  - 密碼強度驗證：長度 ≥ 6，且至少符合 3 種字元類型
- 認證與授權
  - JWT HS256 簽章
  - JWT 具備 exp claim，有效期 24 小時
  - HttpOnly Cookie 傳輸認證資訊
  - `get_current_user()` 驗證登入身份與帳號狀態
  - `require_admin()` 驗證管理員權限
- 前端路由保護
  - GuestOnly
  - RequireAuth
  - RequireAdmin
- 輸入驗證
  - Pydantic v2 強制型別驗證
  - API 請求格式錯誤自動回傳 422
  - Email 格式與角色值限制
- 網路層防護
  - CORS 白名單限制允許來源
  - axios 統一攔截 401
- 可稽核性
  - Thread-safe logger 記錄 11 種敏感操作
  - 管理員可於 AdminLogs 查閱操作歷史

### 講者稿
資安設計不是只做登入，而是從密碼、Cookie、後端授權、輸入驗證到操作日誌都有分層防護。  
其中 HttpOnly Cookie、後端 Depends 授權與帳號鎖定，是目前認證安全的核心。  
Pydantic 驗證與操作日誌則讓資料格式與管理行為更容易追蹤。

## Slide 12｜資安設計 — 現存風險與改善方向

**負責講者：Person C**

### 內容重點
- 高風險
  - `JWT_SECRET` 硬編碼在 `config.py`
  - 改善方向：改用 `os.getenv("JWT_SECRET")` 搭配 `.env`
  - 密碼重設 API 需確認是否套用與註冊相同的強度檢查
- 中風險
  - `sanitizeInput()` 若僅移除 HTML 標籤，防護不足
  - 改善方向：改用 DOMPurify 處理使用者輸入
  - 臨時密碼若使用非密碼學安全亂數，安全性不足
  - 改善方向：改用 `crypto.getRandomValues()` 或後端安全亂數
- 低風險
  - 密碼長度需設定上限，例如 128 字元，避免 bcrypt DoS
  - 後台大量查詢 API 建議加入 limit / offset 分頁
- 正式部署前優先事項
  - Secret 全部環境變數化
  - 資料庫持久化
  - Cookie 參數依部署環境設定 Secure / SameSite
  - 補齊前後端錯誤處理與稽核規則

### 講者稿
這一頁不是在講已經淘汰的架構，而是針對目前最新版系統做風險盤點。  
目前比較重要的是 JWT secret、密碼重設流程與輸入清理機制。  
如果要正式部署，還需要補上環境變數、資料庫與更完整的 Cookie 安全設定。

## Slide 13｜技術亮點 — React Stale Closure Bug 修復

**負責講者：Person D**

### 內容重點
- 問題描述
  - FoodSearch 輸入關鍵字後搜尋結果不正確
  - debounce 延遲搜尋沒有取得最新 keyword
- 根本原因
  - `setTimeout` callback 建立時會捕捉當下 state 快照
  - 後續 keyword 更新不會自動影響已建立的 closure
  - 導致搜尋請求使用到舊資料
- 修法
  1. 新增 `debouncedKeyword` state
  2. 使用 `useEffect([keyword])` 延遲更新 `debouncedKeyword`
  3. cleanup 函式使用 `clearTimeout()` 取消前一次計時
  4. 主搜尋 effect 改依賴 `debouncedKeyword`、分類、熱量範圍與頁碼
- 驗證方式
  - Playwright 輸入「雞胸肉」
  - 確認搜尋結果正確篩出對應食物

### 講者稿
這是開發過程中遇到的真實 React bug。  
問題不是 API 壞掉，而是 React state 與 JavaScript closure 的互動造成搜尋拿到舊值。  
透過 debouncedKeyword 拆分即時輸入值與實際查詢值後，搜尋邏輯就變得穩定。

## Slide 15｜技術亮點 — Thread-safe Logger 與 Playwright E2E 測試

**負責講者：Person D**

### 講者稿
Logger 需要 thread-safe，是因為後端可能同時處理多個請求，如果同時寫檔就可能造成日誌內容交錯。  
Playwright 則是使用真實瀏覽器模擬使用者操作，不只是測函式，而是測整個系統流程。  
目前 38 個案例都通過，代表主要功能與響應式版面都有被驗證。
